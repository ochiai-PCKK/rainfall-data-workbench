# UC ダウンロード自動化 実験結果

## 1. 目的

本書は、モジュール実装へ入る前に行った Playwright 実験の結果を整理し、何が確認できて何が未確認かを明確にするための文書である。

本書の対象は、ログインページの自動入力、OTP 後のパラメータ画面遷移、確認画面、`変換開始`、メールリンク送信要求までとする。


## 2. 実験対象

実験では以下を確認した。

- 保存済み HTML 上での selector 調査
- 本番ログインページでのメールアドレス入力と `ログイン` 押下
- OTP 入力後のパラメータ画面到達
- パラメータ画面での日付入力、日数選択、bbox 設定
- `確認画面` で開く別タブの捕捉
- `変換開始` 後の挙動


## 3. 確認できたこと

### 3.1 ログインページの自動入力は可能

本番ログインページ `https://tools.i-ric.info/login/` には、以下の要素が存在することを確認した。

- `input[type="email"][name="email"]`
- `input[type="submit"][value="ログイン"]`

したがって、メールアドレス入力と `ログイン` 押下までは自動化可能である。


### 3.2 OTP 後に本番パラメータ画面へ遷移する

`manual-login` 実験では、ログイン押下後に OTP を入力すると、最終 URL は `https://tools.i-ric.info/confirm/` となり、タイトルは `Tool for Rain Data　メインページ` であった。

また、`parameter_page_detected` は `true` であり、OTP 完了後に本番のパラメータ画面へ到達していることを確認した。

参照:

- `outputs/uc_download_probe/manual_login_20260312_102931.json`


### 3.3 bbox はハンドル操作以外でも設定できる見込みが高い

保存済み HTML を使った実験では、Leaflet のハンドルをドラッグしなくても、readonly の緯度経度 input を Playwright 側から直接更新できた。

確認できた入力項目は以下である。

- `input[name="south"]`
- `input[name="nouth"]`
- `input[name="west"]`
- `input[name="east"]`

このため、少なくとも UI 操作の代替として、bbox 値の直接設定は実装候補として妥当である。

参照:

- `outputs/uc_download_probe/local_parameter_20260312_100902.json`


### 3.4 大和川流域の実験用 bbox を確定した

大和川流域の基準 bbox は以下である。

- `min_lon=135.43291667`
- `min_lat=34.33633333`
- `max_lon=135.94752778`
- `max_lat=34.78113889`

実験では、各辺に `0.02` 度の余白を足した以下の値を使用した。

- `south=34.31633333`
- `north=34.80113889`
- `west=135.41291667`
- `east=135.96752778`

この値を、実装時の `yamatogawa` プリセットの既定値として扱う。


### 3.5 期間分割は 3 日単位で機械的に生成できる

取得可能開始日は `2010-01-02`、今回の対象終端は `2025-12-31` とし、3 日単位で分割すると、要求件数は `1948` 件となる。

先頭と末尾は以下である。

- 先頭: `2010-01-02` から `2010-01-04`
- 末尾: `2025-12-30` から `2025-12-31`

参照:

- `outputs/uc_download_probe/period_plan_20260312_100850.json`


### 3.6 `確認画面` から `変換開始` までは進められる

実験では、`確認画面` の別タブが開くこと、その画面で `変換開始` を押せることを確認した。

また、`変換開始` 後は確認タブが先に閉じるケースがあり、この挙動は失敗ではなく、要求受理後の正常系として扱うべきことが分かった。


### 3.7 メールリンク送信要求は成立した

実験時の手動確認により、以下を確認している。

- OTP メールは到着し、入力できた
- `変換開始` 後にメールでリンクが到着した

したがって、同一ブラウザセッション内であれば、ログインからメールリンク送信要求までは成立していると判断できる。


## 4. 実験で分かった制約

### 4.1 ブラウザを閉じるとセッション再利用はできない

`request-link` を別プロセスで実行した際、パラメータ画面へ戻れず、セッションが持続しないことが分かった。

このため、以下の方式は本件では主方式にできない。

- ログイン処理を終了してブラウザを閉じる
- 別プロセスから保存済みプロファイルだけで続きを再開する


### 4.2 実装は同一ブラウザセッション前提にする必要がある

本件では、以下を 1 つのブラウザセッション内で連続して処理する必要がある。

- ログインページ表示
- メールアドレス入力
- `ログイン` 押下
- 人手による OTP 入力
- パラメータ入力
- `確認画面`
- `変換開始`

したがって、実装時の主コマンドは、ログインから要求送信までを同一プロセスで扱う構成にする。


## 5. 実装へ向けた判断

今回の実験結果から、以下を実装方針として採用する。

- Playwright を継続採用する
- headed ブラウザを基本とする
- `tools.i-ric.info/login/` を入口とする
- OTP 入力は人手のままとする
- OTP 完了後は同じブラウザで自動継続する
- bbox は `yamatogawa` プリセットを持ち、既定余白 `0.02` 度を与える
- 期間は `2010-01-02` から `2025-12-31` までを 3 日単位でループする
- `確認画面` の閉鎖は正常完了候補として扱う


## 6. 未確認事項

今回の実験では、単発要求の成立までは確認できたが、以下はまだ未確認である。

- 同一セッション内で 2 件以上の要求を連続送信できるか
- 長時間実行中にセッション切れがどの程度発生するか
- メールリンクを開いて ZIP を取得する後段自動化をどう接続するか

このため、次の実装フェーズでは、まず「同一セッションで複数期間を連続送信するループ」の成立を確認しやすい設計を優先する。

# UC ダウンロード自動化 要件定義

## 1. 本書の対象

本書は、実験結果を踏まえて次に実装するモジュールの要件を定義する。

対象フェーズは、以下の 3 段階とする。

- フェーズ 1: ログインページから開始し、OTP 入力後に同一ブラウザセッションで解析雨量のダウンロードリンク送信要求を繰り返し発行する
- フェーズ 2: 人手で開いたメール本文を貼り付け入力し、ダウンロード URL とデータ期間を抽出して保存する
- フェーズ 3: 保存済み URL を使って ZIP を取得し、期間対応を manifest として保存する

本書では、要求送信結果を少なくとも以下の 2 段階で区別する。

- `accepted`
  - `OK` 相当の応答が表示された
  - `dialog` が表示された
- `accepted_candidate`
  - `確認画面` タブが閉じたが、`OK` または `dialog` は観測できていない

`accepted_candidate` は次期間へ進むための暫定成功候補として扱ってよいが、最終集計では `accepted` と分けて記録できなければならない。


## 2. 目的

本機能の目的は、手動では煩雑な解析雨量ダウンロード要求とメール経由ダウンロード運用を整理し、対象期間全体に対して要求送信、メール本文整理、ZIP 保存を追跡可能な形で運用できるようにすることである。

初期実装では、フェーズ 1 を先行実装済みの前提で、フェーズ 2 とフェーズ 3 を追加しても全体の整合性が崩れない構造にすることを重視する。


## 3. スコープ

本機能で対象とするのは以下である。

- ログインページを開く
- メールアドレスを入力する
- `ログイン` を押す
- 人手による OTP 入力を待つ
- パラメータ画面への到達を検知する
- 開始日、取得日数、bbox を設定する
- `確認画面` を開く
- `変換開始` を押す
- 要求受理を検知する
- 次の期間へ進み、これを繰り返す
- 実行ログ、結果 JSON、スクリーンショットを残す
- 人手でコピーしたメール本文の貼り付け入力を受け付ける
- メール本文からダウンロード URL とデータ期間を抽出する
- 抽出した期間が連続しているか、重複や欠落がないかを検査する
- 不整合がある場合に警告し、やり直しまたはスキップを判断できるようにする
- 保存済み URL から ZIP をダウンロードする
- ZIP 保存先と対象期間の対応を manifest として記録する
- どの期間が要求済み、メール取込済み、ZIP 取得済みかを追跡できるようにする

本機能では以下は対象外とする。

- OTP の自動取得
- メール受信箱の自動操作
- メール本文のコピーや貼り付けそのものの自動化
- `uc_rainfall` への自動取り込み


## 4. 対象サイト要件

入口 URL は以下とする。

- `https://tools.i-ric.info/login/`

ログイン後に到達するパラメータ画面の URL は以下を第一候補とする。

- `https://tools.i-ric.info/confirm/`

実装は、固定 URL だけに依存せず、少なくとも以下の画面特徴でパラメータ画面を判定できることが望ましい。

- `#start_day`
- `select[name="days"]`


## 5. セッション要件

本機能は、ログインからリンク送信要求までを同一ブラウザセッションで連続実行できることを必須要件とする。

前提は以下である。

- ブラウザを閉じるとセッション再利用が成立しない可能性が高い
- そのため、ログイン後に別プロセスで再開する方式を主方式にしてはならない

したがって、主たる実装形態は以下とする。

- 1 プロセス
- 1 ブラウザ
- 1 セッション
- OTP 完了後は同じブラウザでそのまま次へ進む


## 6. メールアドレス要件

初期実装で扱うメールアドレスは以下を既定値としてよい。

- `yuuta.ochiai@tk.pacific.co.jp`

ただし、実装では外部から変更可能であることが望ましい。


## 7. 対象領域要件

本機能で対象とする bbox は、大和川流域を覆う設定とする。

基準 bbox は以下である。

- `min_lon=135.43291667`
- `min_lat=34.33633333`
- `max_lon=135.94752778`
- `max_lat=34.78113889`

初期実装では、各辺に `0.02` 度の余白を加えた以下を既定 bbox とする。

- `south=34.31633333`
- `north=34.80113889`
- `west=135.41291667`
- `east=135.96752778`

実装上は、以下の 2 段構成とすることが望ましい。

- 基準 bbox をプリセットとして保持する
- 余白量を別設定として持つ


## 8. 期間要件

本機能で対象とする期間は以下である。

- 開始: `2010-01-02`
- 終了: `2025-12-31`

分割ルールは以下とする。

- 1 回の要求日数は最大 3 日
- 原則として 3 日単位で分割する
- 末尾のみ 1 日または 2 日になることを許容する

この条件では、要求件数は `1948` 件である。


## 9. 入力要件

パラメータ画面で自動設定する値は最低限以下とする。

- 開始日
- 日数 `1` / `2` / `3`
- 緯度南端
- 緯度北端
- 経度西端
- 経度東端

bbox の設定は、地図ハンドルのドラッグではなく、readonly input への直接設定を第一候補とする。


## 10. 機能要件

### 10.1 ログイン開始

- ログインページを開けること
- メールアドレス欄へ値を入力できること
- `ログイン` を押せること


### 10.2 OTP 待機

- 人手による OTP 入力完了を待てること
- パラメータ画面が表示されたことを検知できること
- 指定時間内に到達しない場合は失敗として記録できること


### 10.3 単一期間の要求送信

- 開始日を設定できること
- 日数を設定できること
- bbox を設定できること
- パラメータ画面と確認画面に反映された bbox が期待値と一致するか検証できること
- `確認画面` の別タブを捕捉できること
- 確認画面で `変換開始` を押せること
- 要求送信結果を `accepted` と `accepted_candidate` に分けて記録できること


### 10.4 複数期間のループ実行

- 期間分割計画に従って順番に要求を送れること
- 1 件成功後に次の開始日へ進めること
- ブラウザを閉じずに連続実行できること
- 各要求後に元のパラメータ画面が再利用可能かを確認できること
- 次期間へ進む前に要求間隔または安定待機時間を設定できること
- どの期間まで完了したかを記録できること


### 10.5 結果記録

- 各要求について開始日、終了日、日数、bbox、時刻を記録できること
- 成否を `accepted`, `accepted_candidate`, `failed` として記録できること
- スクリーンショットを必要に応じて保存できること
- 後で再確認できる JSON または CSV を出力できること


### 10.6 メール本文取り込み

- 人手で貼り付けたメール本文を 1 件または複数件まとめて取り込めること
- 本文から少なくとも以下を抽出できること
  - ダウンロード URL
  - データ期間の開始日
  - データ期間の終了日
- 元の本文テキストを後で見返せる形で保存できること
- 抽出結果を構造化データとして保存できること
- 同一 URL または同一期間の重複登録を検知できること


### 10.7 期間整合性チェック

- 取り込んだメール本文群について、期間順に並べたときの連続性を確認できること
- 前件の終了日の翌日が次件の開始日でない場合に警告できること
- 期待期間に対して未取得区間を検出できること
- 期待期間に対して重複区間を検出できること
- 取り込み失敗や欠落があっても、利用者が当該期間を `skipped` として記録して先へ進めること
- 警告が出た場合でも、保存を取り消すか継続するかを判断できること


### 10.8 ZIP 取得と manifest 保存

- 保存済み URL に対して HTTP リクエストを送り ZIP を取得できること
- ZIP を所定ディレクトリへ保存できること
- ファイル名または manifest 上で、ZIP と対象期間の対応を一意に追跡できること
- ZIP 取得成功、取得失敗、スキップを区別して記録できること
- 再実行時に既取得 ZIP を識別し、重複取得を避けられること
- 要求送信結果、メール取込結果、ZIP 保存結果を横断して参照できる manifest を保存できること


## 11. 失敗時の要件

少なくとも以下を区別して記録できることが望ましい。

- ログインページに到達できない
- OTP 完了後にパラメータ画面へ遷移しない
- `確認画面` が開かない
- `変換開始` を押せない
- `accepted` も `accepted_candidate` も取れない
- メール本文から URL または期間を抽出できない
- 期間の連続性が崩れている
- ZIP ダウンロードに失敗する
- ZIP は取得できたが manifest 更新に失敗する

また、異常終了しても、直前に処理していた期間を特定できることが望ましい。


## 12. 非機能要件

- ブラウザは headed 実行を基本とする
- ログ、メッセージ、docstring、コメントは日本語を基本とする
- 実行結果はローカルで追跡可能であること
- 長時間実行を前提に、途中状況を外から確認しやすいこと
- 人手運用の途中参加が可能であること
- テキスト貼り付けを前提としても、再現可能な形で履歴を残せること


## 13. 成功条件

本機能は、以下を満たしたとき成功とする。

- ログインページでメールアドレス入力と `ログイン` 押下が自動で行える
- OTP 入力後に同一ブラウザでパラメータ画面へ到達できる
- 大和川流域 bbox と指定期間を使って単発要求を送れる
- `確認画面` と `変換開始` を通過できる
- メールリンク送信要求の結果を `accepted` と `accepted_candidate` に分けて記録できる
- 同一ブラウザ内で複数期間のループへ拡張できる構造になっている
- 人手で貼り付けたメール本文から URL と期間を安定して抽出できる
- 取り込んだ期間に対して連続性、重複、欠落を判定できる
- 保存した URL から ZIP を取得し、対象期間との対応を manifest に保存できる
- 失敗期間や未取得期間を後で再開可能な形で残せる


## 14. 次段階の論点

本フェーズの次では、主に以下を検討する。

- 1948 件全体を安全に回すループ実装
- 再開機能
- セッション切れ時の自動停止と再ログイン案内
- Outlook など受信箱側の自動補助をどこまで許容するか
- `uc_rainfall` への接続方法

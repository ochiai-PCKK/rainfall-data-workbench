# UC ダウンロード自動化 Overview

## 0. この文書の位置づけ

本書は、実験実施前の段階で作成した前提整理メモである。

そのため、本書には実験前の仮説や未確定事項が含まれる。
現在の確定事項や実装フェーズの判断は、以下の文書を優先する。

- `requirements.md`

特に、セッションの扱い、ログイン後遷移先、確認タブ閉鎖の解釈、実装スコープは、実験後に更新された上記文書を正とする。

## 1. 目的

本機能は、UC の解析雨量ダウンロード画面に対する繰り返し操作を自動化し、`uc_rainfall` がそのまま取り込める ZIP ファイルをローカルへ蓄積できるようにすることを目的とする。

初期段階では、ログインおよびメールワンタイムパスワード入力までは人間が担当し、その後のブラウザ操作、ダウンロード要求、ZIP 保存を自動化対象とする。


## 2. 背景

現状の手動運用には以下の負荷がある。

- 1回のダウンロードで取得できる期間が最大3日間である
- 期間を分割して何度も取得する必要がある
- 対象領域は数値入力ではなく GUI 上のハンドル操作で緯度経度範囲を合わせる必要がある
- 期間指定は開始日入力に加えて、`1日`, `2日`, `3日` のプルダウン選択が必要である
- 要求確定の途中で別タブが開く
- 最終確定後にメールでダウンロードリンクが発行される

このため、人が毎回同じ操作を繰り返す運用は手間が大きく、期間分割数が増えるほど負荷が高くなる。


## 3. 現行フロー

現時点で把握している操作フローは以下の通りである。

1. 利用者がメールアドレスを入力する
2. メールで届いたワンタイムパスワードを入力してログインする
3. 解析雨量ダウンロード画面で対象範囲を GUI ハンドル操作で調整する
4. 開始日を入力する
5. 取得日数を `1`, `2`, `3` のいずれかからプルダウンで選択する
6. ダウンロード要求を確定する
7. 別タブが開く
8. 新しいタブ側の確認画面で `変換開始` を押す
9. 変換開始後に上部から `OK` ボタン付きの通知が表示される
10. `OK` を押すと確認タブが閉じる
11. メールにダウンロードリンクが発行される
12. そのリンクを開くと ZIP をダウンロードできる

このフローのうち、初期実装では 1 と 2 を手動、3 以降を自動化候補として扱う。


## 4. 自動化の狙い

自動化で解決したい主な課題は以下である。

- 大和川流域を覆う範囲合わせを毎回手で行わなくてよくすること
- 最大3日間制約に合わせた繰り返し要求を自動実行できること
- ダウンロード済み ZIP を整理されたディレクトリへ保存できること
- 後続の `uc_rainfall` 取り込み処理へそのまま渡せる形にすること
- 実行ログと結果一覧を残し、失敗時にどこで止まったか追跡できること


## 5. 初期スコープ

初期スコープでは以下を対象とする。

- ログイン済みブラウザ状態を引き継いで処理を開始できること
- 解析雨量ダウンロード画面の操作を自動化できること
- 期間分割ルールに従って複数回の要求を実行できること
- ZIP ダウンロード完了を検知し、所定ディレクトリへ保存できること
- ダウンロード結果を manifest やログとして保存できること
- `uc_rainfall` とは独立したパッケージとして開発できること

初期スコープでは以下は対象外とする。

- メール受信箱そのものの自動操作
- OTP 取得の自動化
- サイト内部 API の直接呼び出しへの依存
- `uc_rainfall` CLI との一体化


## 6. 技術方針

初期の第一候補は、Python から利用する Playwright ベースのブラウザ自動化とする。

理由は以下の通りである。

- GUI 操作主体の画面に対してブラウザ操作を安定して書きやすい
- タブ遷移とダウンロード待機を扱いやすい
- 人手ログイン後のセッションを専用プロファイルで保持しやすい
- 失敗時にスクリーンショット、トレース、ログを残しやすい

想定する実装方針は以下である。

- `src` 配下に新しい独立パッケージを作る
- パッケージ名の第一候補は `uc_download` とする
- ブラウザは headless ではなく headed を基本とする
- Playwright の persistent context を用いて、手動ログイン後の状態を引き継ぐ
- ZIP 保存先、ダウンロード単位、対象期間、実行結果は設定ファイルまたは manifest に記録する


## 7. `uc_rainfall` との関係

既存の `uc_rainfall` は、展開済みディレクトリまたは ZIP ファイルを入力として扱える構成になっている。

そのため、今回の新パッケージは以下の責務に限定するのが自然である。

- ダウンロード対象期間を決める
- ブラウザ操作で要求を発行する
- ZIP をローカルへ保存する
- 保存した ZIP 一覧を後続処理へ渡せる形にまとめる

これにより、取得処理と解析処理の責務を分離できる。


## 8. 開発用画面素材

開発用資料として、`docs/uc_download_automation/sourse_page` に対象画面の保存済み HTML と同梱リソースを置く。

含まれる主な素材は以下である。

- `UC Tool メニュー.html`
- `Tool for Rain Data　メインページ.html`
- `Tool for Rain Data　メインページパラメータ入力.html`
- `Tool for Rain Data　確認画面.html`

これらの素材により、少なくとも以下を事前に確認できる。

- 開始日入力欄の id や name
- 取得日数プルダウンの構造
- 確認画面の form 構造
- 別タブ遷移の有無
- Leaflet area select を使った矩形選択 UI の存在

ただし、保存済み HTML だけでは以下は確定できない。

- ログイン後の実サイトで動作する JavaScript による DOM 変化
- `OK` 通知の実際の DOM 構造
- メール送信完了後の実レスポンス
- セッション切れ時の挙動


## 9. 想定成果物

本機能の初期成果物は以下を想定する。

- ローカル保存された ZIP ファイル群
- 各 ZIP に対応する期間や取得結果をまとめた manifest
- 実行ログ
- 障害調査用スクリーンショットまたはトレース


## 10. 設計上の主要論点

現時点で特に整理が必要な論点は以下である。

- GUI ハンドル操作で対象範囲をどのように安定再現するか
- 別タブ遷移後の確定操作をどのように待機・検知するか
- `変換開始` 後に出る `OK` 通知をどのように検知するか
- メールリンク開封を初期段階でどこまで自動化するか
- 3日単位の期間分割ルールをどこで管理するか
- ダウンロード済み期間の重複取得をどう避けるか
- セッション期限切れ時の再ログインをどのように扱うか
- 画面変更に弱いセレクタをどう管理するか

# UC降雨処理 要件定義

## 1. 目的

本機能は、iRIC-UC の UC-tools からダウンロードした取得済み降雨データを入力として、時系列化、セル選定、空間対応付け、時間雨量および累加雨量の可視化までを行うことを目的とする。

対象は本リポジトリで扱うデータに限定し、既存の `Water-Info-Acquirer` の実装は参考とするが、その入力前提や内部構造には依存しない。


## 2. スコープ

本要件では、UC-tools からダウンロードした降雨データに対する処理機能のうち以下を対象とする。

- 入力データからグラフ作成に必要な時系列を構築すること
- 入力データを後続利用可能な中間データとして蓄積すること
- 1時間雨量を基準に、複数時間幅の累加雨量を計算すること
- 計算結果をもとにグラフ画像を出力すること
- セル選定および将来の空間集約に利用できる空間対応情報を整備すること
- 蓄積された中間データから対象期間を指定して処理できること
- 同一格子定義を持つ複数 `dataset_id` を、可視化時に連続系列として扱えること
- 同一格子定義の重複登録に対して、完全一致と不整合を識別できること
- 短期間データでも破綻しない形でグラフを生成すること

以下は本要件の対象外とする。

- JAXA API からの新規取得処理
- JAXA 由来データのグラフ作成要件
- 年単位の完全性評価
- 長期統計値の正当性保証
- GUI 実装
- Web フロントエンド実装


## 3. 想定入力

想定入力は、UC-tools からダウンロードした1時間間隔の降雨データを含むローカルデータ一式とする。

現時点では以下の前提を置く。

- 1つの入力データセットは最大でも約3日間程度の連続データを持つ
- 入力は展開済みディレクトリまたは ZIP ファイルのどちらでも指定できることが望ましい
- 観測時刻はファイル名や付帯情報から復元できる
- 本機能内で、時刻列と1時間雨量列を持つ時系列データへ整形できること
- 代表的な入力は、`rain.dat`、関連ラスタ、メール本文等を含む UC-tools のダウンロード成果物である


## 4. 中間データ要件

本機能は、グラフ生成のための一時処理だけで終わらず、後続の可視化や再利用に耐える中間データを保持できることを要件とする。

要件は以下の通り。

- 入力データを Python で解析し、中間データストアへ登録できること
- 中間データには時系列データを継続的に追加できること
- 中間データから再取り込みなしにグラフ生成へ利用できること
- 将来的に API 経由でフロントエンドから参照できる構造であること
- 生データと集約済みデータを区別して保持できること
- 完全一致のデータセットは重複登録せずスキップできること
- 重複時刻で値が異なるデータセットは不整合として検出できること

本機能の処理は、少なくとも以下の2フェーズで構成できることを要件とする。

- フェーズ1: 入力解析、時刻復元、空間判定、中間データ格納
- フェーズ2: 中間データ参照、対象セル選択、累加雨量計算、グラフ生成

フェーズ2では、ユーザーが表示対象期間を指定できることを要件とする。


## 5. 空間領域要件

本機能では、処理対象となる空間領域を以下の4種類とする。

- 大和川流域
- 東除川流域
- 西除川流域
- 東除川流域 + 西除川流域

各領域はポリゴンとして保持し、各ポリゴンに対応する BBox を管理対象とする。

セル選定は以下の二段階で行えることを要件とする。

- BBox により候補セルを粗く絞り込むこと
- ポリゴンとの空間判定により対象セルを確定すること

BBox のみで最終判定してはならない。


## 6. 座標系要件

本機能における解析用の座標系は、流域ポリゴンが保持する CRS を基準とする。

要件は以下の通り。

- 解析時は、ラスタセルやその代表点をポリゴン側 CRS にそろえて扱えること
- 流域ポリゴンの CRS を空間判定の基準とすること
- 生データの元座標系情報は保持できること
- BBox 計算およびセル選定は、解析対象の CRS 上で整合が取れていること


## 7. セル選定要件

どのセルを処理対象およびグラフ化対象とするかを明確に扱えることを要件とする。

最低限、以下のような選定処理を実現可能な構造とする。

- 領域ポリゴン内に入るセルの抽出
- ある領域に属するセル一覧の保持
- 領域ごとの代表時系列作成に向けたセル集合の確定

初期実装では、グラフ対象セルは空間判定で抽出されたセル集合の中から、候補ビューを介してユーザーが指定する方式とする。
初期実装では、セル単位グラフに加えて、流域内全セルの合計および平均によるグラフを出力できることを要件とする。

候補セルの内部識別には `row` および `col` を用いることを基本とする。
初期実装の候補ビューは、少なくとも一覧形式で候補セルを確認できることを要件とする。
候補ビューには、少なくとも `polygon_name`, `row`, `col`, `x_center`, `y_center`, `inside_flag` を表示できることが望ましい。

セルサイズは固定である前提を置くが、基準座標およびセル数はデータセットごとに異なり得るものとして扱う。


## 8. 出力要件

本機能は、空間対応情報、中間データ、および降雨グラフ画像を出力できることを要件とする。

最低限必要な出力は以下とする。

- 中間データストア
- セル選定結果または空間対応情報
- 1時間雨量を棒グラフで表示した画像
- 累加雨量を折れ線で重ねた画像

初期実装では、グラフ出力単位は `1セル × 1指標 = 1画像` を基本とする。
したがって、1回の処理で最大6枚の画像を出力し得る。
流域集計モードでは、`1流域 × 1集計方法 × 1指標 = 1画像` を基本とする。

出力形式は PNG を第一候補とする。


## 9. 計算対象

本機能では、1時間雨量を基準に以下の累加雨量を計算対象とする。

- 1時間
- 3時間
- 6時間
- 12時間
- 24時間
- 48時間

48時間累加雨量は、連続48時間分の1時間データが存在する場合に計算可能とする。


## 10. 期間指定要件

本機能は、蓄積済み中間データに対して表示対象期間を指定できることを要件とする。

要件は以下の通り。

- ユーザーが表示開始時刻と表示終了時刻を指定できること
- 指定期間内のデータだけをグラフ表示対象にできること
- 累加雨量計算に必要な過去データは内部的に追加参照してよいこと
- 表示対象期間と内部計算期間を区別して扱えること
- 同一格子定義を持つ複数 `dataset_id` が存在する場合、それらを結合した上で表示対象期間を評価できること


## 11. 短期データに対する取り扱い

入力データが最大3日間程度であることを踏まえ、以下を要件とする。

- 3日間データでもグラフ生成できること
- 48時間累加雨量が計算可能な場合は、その値を扱えること
- データ長が不足する時間幅については、計算不能な先頭区間を欠損として扱うこと
- データ期間外の値を補完してはならない
- 出力される最大値は「入力期間内での最大値」として扱うこと


## 12. 時刻に関する要件

本機能は、入力データに対応する明示的な時刻列を内部で扱う。

要件は以下の通り。

- 1時間間隔の連続時系列として扱えること
- 時刻情報がファイル名等から復元できる場合は、それを優先して採用すること
- 入力データの時刻基準は JST として扱うこと
- 中間データストアへの保存基準も JST とすること
- 時刻が欠けている場合でも、欠測として明示的に扱えること


## 13. 欠測データの扱い

欠測を含むデータでも処理継続できることを要件とする。

- 欠測値は `NaN` 等の明示的な欠損として保持すること
- 欠測をゼロ雨量として自動補完してはならない
- 累加雨量の計算では、必要時間幅を満たさない区間を欠損として扱うこと
- グラフ上で欠測が識別可能であることが望ましい


## 14. グラフ表現要件

グラフは、短期間の降雨イベントを把握しやすいことを重視する。

- 左軸に1時間雨量を表示すること
- 右軸に累加雨量を表示すること
- 横軸は時刻とすること
- 指定期間全体をそのまま描くのではなく、指定期間内の各指標最大イベントを切り出して表示できること
- 対象時間幅ごとに見やすい表示範囲を選べること
- 3日分程度の短期データでも読める軸密度で描画できること

最大値が同値で複数存在する場合は、最初の発生時刻を採用する。
採用しなかった他の同値発生時刻は、確認可能な形でログ出力できることが望ましい。


## 15. 成功条件

本機能は、以下を満たしたとき要件を満たすものとする。

- 入力データが中間データストアへ蓄積できる
- ローカル入力データから時系列を構築できる
- セル選定に必要な空間対応情報を生成できる
- ユーザーが候補ビューを通じて候補セルからグラフ対象セルを指定できる
- ユーザーが表示対象期間を指定できる
- 同一格子定義の複数 `dataset_id` を連結してグラフ生成できる
- 1時間から48時間までの累加雨量を算出できる
- 3日間程度のデータでも破綻せずグラフを出力できる
- 出力グラフから降雨の時間変化と累加傾向を確認できる


## 16. 今後の設計論点

設計フェーズでは、特に以下を整理する。

- 中間データストアを SQLite にするか他方式にするか
- API 経由での参照を見据えたデータ構造にするか
- フェーズ1とフェーズ2の責務境界をどこまで明示するか
- 入力データ形式ごとの時刻抽出方法
- 4領域をどのデータ構造で管理するか
- 空間対応情報をどの粒度で保存するか
- セル採用判定をセル中心判定にするか、重なり率判定にするか
- 候補セルビューでどの属性を見せるか
- 表示対象期間の UI/API 指定方法
- `polygon_timeseries` を将来拡張としてのみ管理するか
- グラフを1枚にまとめるか、時間幅ごとに分けるか
- 欠測の可視化方法
- 最大イベント抽出の定義
- 出力ファイル命名規則

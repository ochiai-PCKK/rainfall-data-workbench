# UC降雨処理 設計

## 1. 設計方針

本機能は、UC-tools からダウンロードした降雨データを直接グラフ化するのではなく、いったん解析向けの中間データへ正規化した上で、空間対応付けとグラフ生成へ利用する。

この構成により、以下を実現する。

- 入力形式依存の処理とグラフ処理を分離する
- セル単位データと将来の流域集約データを同じ基盤で扱えるようにする
- 空間判定ロジックを後から差し替えやすくする
- JAXA 等の別ソースを将来追加するときも出口をそろえやすくする
- 将来的に API やフロントエンドから再利用しやすい形でデータを保持する


## 2. 全体アーキテクチャ

処理は以下の2フェーズ5段階に分ける。

フェーズ1: 取り込み・空間対応・中間データ格納

フェーズ2: 中間データ参照・可視化

1. 入力解析
2. 中間データ生成
3. 空間対応付け
4. グラフ生成
5. 将来の API / Frontend 再利用

処理フローは以下を想定する。

- UC-tools ダウンロードディレクトリを読み取る
- `rain.dat`、関連ラスタ名、メール本文等から時刻と格子定義を抽出する
- セル単位の時系列データを中間データへ保存する
- ポリゴンを基準 CRS で読み込み、BBox とセル対応を計算し、その結果を中間データへ保存する
- 中間データを参照し、空間判定で抽出されたセル集合から候補ビューを介してユーザー指定でグラフ対象セルを選ぶ
- ユーザー指定の表示期間を受け取り、必要な内部計算期間を含めて時系列を取得する
- 選択したセルの累加雨量を計算し、表示期間に切り出して PNG グラフを出力する
- 将来的には中間データを API 経由でフロントエンドへ提供する


## 3. コンポーネント分割

本機能は、最低限以下の責務に分割する。

### 3.1 入力ローダ

責務:

- UC-tools ダウンロードディレクトリを受け取る
- `rain.dat` の構造を読み取る
- 関連ファイル名から観測時刻候補を抽出する
- メール本文等の補助情報を取得する

出力:

- データセット情報
- 格子定義情報
- セルごとの生時系列


### 3.2 時刻復元器

責務:

- ファイル名から JST 時刻を抽出する
- 必要なら UTC 表現との対応を保持する
- `rain.dat` ブロック先頭値を経過秒として解釈する
- 入力期間に対して整合する観測時刻列を確定する
- 中間データストアへ保存する JST 時刻を確定する

時刻決定優先順位は以下とする。

1. ファイル名中の `_JST_YYYYMMDD_HHMMSS`
2. UTC 相当ファイル名からの変換
3. `rain.dat` ブロック先頭値の経過秒
4. メール本文の期間情報


### 3.3 空間定義ローダ

責務:

- 流域ポリゴンを読み込む
- 基準とする4領域を管理する
- 各領域の BBox を保持する
- 解析用 CRS を確定する

対象領域:

- 大和川流域
- 東除川流域
- 西除川流域
- 東除川流域 + 西除川流域


### 3.4 セル空間対応付け

責務:

- 格子セルの位置を座標化する
- BBox で候補セルを絞り込む
- ポリゴンとの空間判定で対象セルを確定する
- 領域ごとのセル集合を保存する

本コンポーネントの出力はフェーズ1で DB へ格納する主要成果物の一つとする。

初期実装では、セル中心点がポリゴン内にあるかで判定する方式を第一候補とする。

理由:

- 実装が単純
- セルサイズ一定前提と相性がよい
- 重なり率判定へ後から拡張しやすい


### 3.5 時系列集約器

責務:

- 領域に属するセル群を取得する
- 将来的にセル単位時系列から領域代表時系列を生成する
- 将来的に欠測を保持したまま集約する

初期実装では、流域代表値の生成は実装対象外とする。

将来拡張候補:

- 面積重み平均
- 最大値
- セル単位個別グラフ


### 3.6 グラフ生成器

責務:

- 将来的には領域代表時系列から 1h, 3h, 6h, 12h, 24h, 48h の累加雨量を計算する
- 初期実装ではセル単位時系列を入力として 1h, 3h, 6h, 12h, 24h, 48h の累加雨量を計算する
- ユーザー指定の表示期間を受け取り、内部計算期間を補正する
- 指定期間内の各指標最大イベントを抽出する
- 同値最大が複数ある場合は最初の発生時刻を採用し、他候補をログ出力する
- 指定時間幅ごとにグラフを描画する
- PNG を出力する

初期実装では、グラフ対象セルは `polygon_cell_map` に登録されたセル集合から候補ビューを介してユーザー指定で選択する。
内部識別子には `row` および `col` を用いる。
候補ビューは初期実装では一覧形式を第一候補とする。
一覧には少なくとも `polygon_name`, `row`, `col`, `x_center`, `y_center`, `inside_flag` を表示する。

参考実装は `Water-Info-Acquirer` の `outputs/analysis.py` と `outputs/chart_exporter.py` とするが、本リポジトリでは入力前提に合わせて独立実装する。


### 3.7 将来 API 提供層

責務:

- 中間データストアから必要な時系列を取り出す
- 領域、期間、指標を条件にデータを返す
- フロントエンド向けのレスポンス形式を提供する

初期実装では API 自体は実装対象外とするが、データ構造は API 提供しやすい形を前提に設計する。


## 4. データモデル設計

初期実装では、SQLite を第一候補とする中間データストアを採用する。

理由:

- ローカルで扱いやすい
- 構造が明確
- デバッグしやすい
- 後段の検索と集約が簡単
- Python による取り込み処理と相性がよい
- 将来バックエンド API のデータソースとして利用しやすい

DB への主要な格納処理はフェーズ1で実施する。

### 4.1 datasets

データセット単位のメタ情報を保持する。

想定カラム:

- `dataset_id`
- `source_type`
- `source_dir`
- `time_start`
- `time_end`
- `crs_raw`
- `created_at`


### 4.2 grids

格子定義を保持する。

想定カラム:

- `dataset_id`
- `grid_crs`
- `origin_x`
- `origin_y`
- `cell_width`
- `cell_height`
- `rows`
- `cols`


### 4.3 cell_timeseries

セルごとの時系列を保持する。

想定カラム:

- `dataset_id`
- `observed_at`
- `row`
- `col`
- `x_center`
- `y_center`
- `rainfall_mm`
- `quality`


### 4.4 polygons

対象流域ポリゴンと領域メタ情報を保持する。

想定カラム:

- `polygon_id`
- `polygon_name`
- `polygon_group`
- `polygon_crs`
- `minx`
- `miny`
- `maxx`
- `maxy`


### 4.5 polygon_cell_map

流域とセルの対応関係を保持する。

想定カラム:

- `dataset_id`
- `polygon_id`
- `row`
- `col`
- `inside_flag`
- `selection_method`


### 4.6 API 利用を見据えた考え方

中間データストアは、単なる一時保存先ではなく、将来の可視化基盤のデータソースとして扱う。

そのため以下を満たすように設計する。

- 領域単位で時系列を取得しやすいこと
- 期間条件で抽出しやすいこと
- 生データと集約済みデータを分離できること
- 再計算が不要な出力を優先的に保存できること

初期実装では、`polygon_timeseries` は将来拡張用の設計要素とし、初期 DB スキーマには含めない。

想定する将来 API 例:

- `GET /datasets`
- `GET /polygons`
- `GET /cell-timeseries?dataset_id=...&row=...&col=...&start=...&end=...`
- `GET /polygon-timeseries?polygon_id=...&start=...&end=...`


## 5. 座標系設計

解析時の基準 CRS は流域ポリゴン側の CRS とする。

処理方針は以下とする。

- 入力データの元 CRS 情報は保持する
- 格子セル中心座標は必要に応じて解析 CRS に変換する
- ポリゴンとの空間判定は解析 CRS 上で行う
- BBox も解析 CRS 上で保持する

実装上は以下の順序を採る。

1. ポリゴンを読み込む
2. 基準 CRS を決める
3. 格子定義を同一 CRS 上へ写す
4. セル中心点またはセル形状を生成する
5. 空間判定を実施する


## 6. セル選定ロジック設計

初期実装では、以下の二段階判定を採用する。

### 6.1 候補抽出

- 領域 BBox 内に入るセルのみを候補とする

### 6.2 確定判定

- セル中心点がポリゴン内にあるセルを対象セルとする

初期実装では、この確定判定結果を DB に保存し、後続のグラフ生成では DB から参照する。

この方式の利点は以下の通り。

- 計算量を抑えやすい
- 実装が単純
- セルサイズ一定前提に適する

制約は以下の通り。

- 境界付近のセルは実際の重なり面積を反映しない

そのため、将来は以下への拡張を可能とする設計にする。

- セルとポリゴンの重なり率判定
- 閾値ベース採用
- 面積重み集約


## 7. 時系列設計

基本時系列は 1時間雨量とする。

内部表現は以下を前提とする。

- `observed_at`
- `rainfall_mm`
- `quality`

`observed_at` は入力・保存ともに JST 基準とする。

累加雨量はグラフ用派生系列として計算する。

対象:

- 1時間
- 3時間
- 6時間
- 12時間
- 24時間
- 48時間

計算ルール:

- rolling sum を用いる
- 必要時間幅を満たさない区間は欠測とする
- 欠測を 0 に置換しない

初期実装では、rolling 系列は DB に保存せず、グラフ生成時にその場で計算する。

表示期間と内部計算期間は分離する。

例:

- 表示期間: `view_start` から `view_end`
- 内部計算期間: `calc_start` から `view_end`

`calc_start` は必要な累加時間幅に応じて `view_start` より前へ拡張する。
48時間累加を表示する場合、少なくとも 48時間分の過去データを内部計算に用いる。


## 8. 欠測処理設計

欠測は明示的に保持する。

方針:

- 時刻列は連続 1時間間隔で保持する
- データが存在しない時刻は欠測行として表現できるようにする
- 集約時も欠測情報を失わない
- グラフ上で欠測が視認可能な表現を採用できるようにする


## 9. グラフ設計

初期実装では、選択したセル単位時系列を対象としたイベント把握用グラフを出力する。

表示要素:

- 左軸: 1時間雨量の棒グラフ
- 右軸: 指定時間幅の累加雨量の折れ線
- 横軸: 観測時刻

対象時間幅:

- 1時間
- 3時間
- 6時間
- 12時間
- 24時間
- 48時間

初期実装では、ユーザー指定期間全体をそのまま1枚に描くのではなく、指定期間内の各指標最大イベントを切り出して描画する。

最大イベントの決定ルール:

- 指定期間内で各指標の最大値を求める
- 最大値が同値で複数ある場合は最初の発生時刻を採用する
- 採用しなかった他の同値発生時刻はコンソールログ等へ出力する

切り出し幅は `Water-Info-Acquirer` の参考実装にある指標別設定を準用する。
すなわち、1時間から48時間までの各指標ごとに、前後の表示幅および軸ラベル間隔を個別に持つ。


## 10. 出力設計

出力物は最低限以下とする。

- 中間データストア
- セル単位時系列
- セル選定結果または空間対応情報
- PNG グラフ

フェーズ1の主出力:

- `datasets`
- `grids`
- `cell_timeseries`
- `polygons`
- `polygon_cell_map`

フェーズ2の主出力:

- 選択セルに対するグラフ画像

初期実装のグラフ出力単位は `1セル × 1指標 = 1画像` とする。

ファイル命名規則は設計詳細で別途確定するが、最低限以下を含める。

- データセット識別子
- 領域名
- 指標名

初期実装では、以下のような命名規則を第一候補とする。

`{dataset_id}_{polygon_name}_r{row}_c{col}_{metric}_{event_time_jst}.png`

将来的には、同じ中間データを API やフロントエンドから再利用する前提で出力を管理する。


## 11. 初期実装の制約

初期実装では、以下の制約を受け入れる。

- 対象ソースは UC-tools ダウンロードデータのみ
- 入力期間は短期イベント中心
- セル選定は中心点判定ベース
- グラフ対象セルは候補セル集合からのユーザー指定
- 時刻保存基準は JST 固定
- 流域代表時系列の生成は初期実装対象外
- `polygon_timeseries` は将来拡張概念とし、初期 DB スキーマには含めない
- rolling 系列は初期実装では DB 保存せず、グラフ生成時に計算する
- 可視化は静的 PNG 出力
- API / フロントエンド連携は将来拡張とする


## 12. 将来拡張

将来は以下を拡張候補とする。

- JAXA データの同一基盤への統合
- 重なり率ベースのセル選定
- 面積重み集約
- `polygon_timeseries` テーブルの追加
- DB からの動的可視化
- API 提供層の実装
- セル単位と流域単位の切り替え
- GUI または Web UI からの閲覧

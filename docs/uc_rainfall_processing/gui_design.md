# UC降雨処理 GUI設計

## 1. 目的

本書は、既存の UC 降雨処理 CLI / サービス層を利用して、ローカル GUI を実装するための設計を定義する。

初期実装では Tkinter を採用し、以下を GUI から操作できることを目的とする。

- DB 初期化
- ZIP または展開済みディレクトリの取り込み
- 候補セル一覧の確認
- セル単位 / 流域集計のグラフ出力
- 実行ログの確認


## 2. 前提

GUI は既存のサービス層を直接呼び出す。

利用対象:

- `ingest_uc_rainfall()`
- `ingest_uc_rainfall_many()`
- `list_candidate_cells()`
- `generate_metric_event_charts()`
- `load_settings()`
- `update_settings()`

初期実装では Web API は挟まない。
設定キャッシュはプロジェクト直下の `.uc_rainfall_settings.json` を利用する。


## 3. 採用方針

### 3.1 技術選定

- GUI フレームワーク: Tkinter
- 理由:
  - ローカルツールとして十分
  - 既存環境で追加依存が少ない
  - CLI / サービス層をそのまま呼びやすい


### 3.2 操作単位

GUI は以下の3機能を中心に構成する。

- 取り込み
- 候補セル確認
- グラフ出力


## 4. 画面構成

初期実装では、単一ウィンドウ内に3つの主要エリアを持つ。

1. 入力エリア
2. 候補セルエリア
3. 実行ログエリア

GUI 起動時には設定キャッシュを読み込み、前回入力値を初期表示へ反映する。


### 4.1 入力エリア

入力項目:

- DB パス
- 入力パス
  - ZIP
  - 展開済みディレクトリ
  - 複数選択可能
- ポリゴンディレクトリ
  - 任意
  - 未指定時は DB 登録済みポリゴンを使用
- 取り込み用 `dataset_id`
  - 任意
  - 単一入力時のみ有効
  - 入力 ZIP / ディレクトリ名とは異なる識別子を付けたい場合に使う
- グラフ用優先 `dataset_id`
  - 任意
  - 未指定時は位置一致する全データセットを対象にする
  - 指定時は同一位置セルのうち優先データセットとして扱う
- 流域名
- 系列モード
  - `cell`
  - `polygon_sum`
  - `polygon_mean`
  - `polygon_weighted_sum`
  - `polygon_weighted_mean`
- 流域内行
- 流域内列
- 表示開始日時
- 表示終了日時
- 出力ディレクトリ

優先的に前回値を復元する項目:

- DB パス
- ポリゴンディレクトリ
- 入力パス一覧
- 出力ディレクトリ
- グラフ用優先 `dataset_id`
- 流域名
- 系列モード
- 表示開始日時
- 表示終了日時

ボタン:

- `DB初期化`
- `取り込み`
- `候補セル一覧更新`
- `グラフ出力`


### 4.2 候補セルエリア

候補セル一覧はテーブル形式で表示する。

表示列:

- 流域名
- 流域内行
- 流域内列
- 中心X
- 中心Y
- 重なり率
- データセット数

操作:

- 行選択
- 選択行から `流域内行` / `流域内列` を入力欄へ反映

初期実装ではソートとフィルタは最小限とし、将来拡張とする。


### 4.3 実行ログエリア

テキストボックス形式でログを表示する。

表示対象:

- 取り込み開始 / 終了
- ポリゴン読込元
  - ファイル
  - DB
- 重複スキップ
- 不整合エラー
- 候補セル数
- 使用データセット
- 最大イベント抽出結果
- 出力ファイルパス


## 5. レイアウト案

推奨レイアウト:

- 上段: ファイル / 実行パラメータ入力
- 中段: 候補セルテーブル
- 下段: ログ

横幅の大きい入力項目:

- DB パス
- 入力パス
- ポリゴンディレクトリ
- 出力ディレクトリ

これらは `Entry + 参照ボタン` とする。
入力パスは複数選択に対応し、選択済み一覧を別ウィジェットで確認できるようにする。


## 6. 画面状態制御

### 6.1 系列モードによる制御

- `cell` の場合
  - `流域内行`
  - `流域内列`
  を有効にする

- `polygon_sum`, `polygon_mean`, `polygon_weighted_sum`, `polygon_weighted_mean` の場合
  - `流域内行`
  - `流域内列`
  を無効にする

### 6.2 入力件数による制御

- 入力パスが 1 件のとき
  - 取り込み用 `dataset_id` を有効にする

- 入力パスが複数件のとき
  - 取り込み用 `dataset_id` を無効にする
  - 自動的に各入力ファイル名 / ディレクトリ名ベースで `dataset_id` を決定する


### 6.2 ポリゴンディレクトリによる制御

- 入力時に `ポリゴンディレクトリ` が空でも実行可能
- ただし DB にポリゴン未登録ならエラー表示


## 7. 操作フロー

### 7.1 初回利用

1. DB パスを指定
2. ポリゴンディレクトリを指定
3. 入力パスを指定
4. 必要なら取り込み用 `dataset_id` を指定
5. `取り込み` を実行
6. `候補セル一覧更新` を実行
7. 候補セルを選択
8. 表示期間と出力先を指定
9. `グラフ出力` を実行


### 7.2 DB 登録済みポリゴン利用

1. DB パスを指定
2. 入力パスのみ指定
3. `取り込み` を実行


### 7.3 既存 DB からのグラフ出力のみ

1. DB パスを指定
2. 流域名を選択
3. `候補セル一覧更新`
4. セルまたは流域集計モードを選択
5. 表示期間と出力先を指定
6. `グラフ出力`


## 8. イベントハンドラ設計

### 8.1 DB 初期化

呼び出し先:

- `initialize_schema()`

結果:

- 成功時にログ出力
- 失敗時はエラーダイアログ
- 成功時は DB パスを設定キャッシュへ保存


### 8.2 取り込み

呼び出し先:

- 単一入力時:
  - `ingest_uc_rainfall()`
- 複数入力時:
  - `ingest_uc_rainfall_many()`

結果:

- 成功時にログ出力
- 重複完全一致時はスキップログ
- 不整合時はエラーダイアログ
- 成功時は入力関連設定を設定キャッシュへ保存

実行ルール:

- 入力 1 件:
  - 取り込み用 `dataset_id` をそのまま渡せる
- 入力 複数件:
  - `ingest_uc_rainfall_many()` を呼ぶ
  - 各 `dataset_id` は入力名から自動決定する


### 8.3 候補セル一覧更新

呼び出し先:

- `list_candidate_cells()`

結果:

- テーブル再描画
- 候補セル 0 件なら通知


### 8.4 グラフ出力

呼び出し先:

- `generate_metric_event_charts()`

結果:

- 出力ファイル一覧をログ表示
- 失敗時はエラーダイアログ
- 成功時は表示期間・流域・出力先を設定キャッシュへ保存

実行ルール:

- グラフ用優先 `dataset_id` 指定あり:
  - `generate_metric_event_charts()` へ `dataset_id` を渡す
  - 同一位置セルのうち優先データセットとして扱う

- グラフ用優先 `dataset_id` 指定なし:
  - `dataset_id=None` で `generate_metric_event_charts()` を呼ぶ
  - 位置一致する全データセットを対象にする


## 9. バリデーション

最低限の入力検証:

- DB パスが空でない
- 入力パスが存在する
  - 取り込み時のみ
- 複数入力時は取り込み用 `dataset_id` を空にする
- グラフ出力時は `polygon_name` が選択済み
- `cell` モードでは `流域内行` / `流域内列` が入力済み
- `view_start <= view_end`
- 出力ディレクトリが指定されている
  - グラフ出力時


## 10. 候補セルテーブルの実装方針

Tkinter の `ttk.Treeview` を第一候補とする。

理由:

- 標準で利用可能
- テーブル表示に十分
- 行選択イベントを扱いやすい

将来拡張:

- 列ソート
- CSV エクスポート
- ダブルクリックでセル選択


## 11. ログの扱い

サービス層は既に日本語ログを出すため、GUI では以下のいずれかを採る。

- 標準 logging を GUI テキストへ転送する
- GUI 側で service 呼び出し結果を補足して追記する

初期実装では、logging ハンドラを追加して GUI テキストへ流す方式を第一候補とする。


## 12. スレッド / 応答性

長時間処理:

- 取り込み
- グラフ出力

これらは UI スレッドを塞ぎやすい。

初期実装では、少なくとも以下を行う。

- 実行中はボタンを無効化
- カーソルまたは状態ラベルで実行中を示す

将来拡張:

- バックグラウンドスレッド実行
- 進捗表示

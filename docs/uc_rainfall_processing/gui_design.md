# UC降雨処理 GUI設計

## 1. 目的

本書は、既存の UC 降雨処理 CLI / サービス層を利用して、ローカル GUI を実装するための設計を定義する。

初期実装では Tkinter を採用し、以下を GUI から操作できることを目的とする。

- DB 初期化
- ZIP または展開済みディレクトリの取り込み
- 候補セル一覧の確認
- 面的可視化ビューによるセル分布確認
- セル単位 / 流域集計のグラフ出力
- 実行ログの確認
- AI テストモードによる GUI 操作と UX 確認


## 2. 前提

GUI は既存のサービス層を直接呼び出す。

利用対象:

- `ingest_uc_rainfall()`
- `ingest_uc_rainfall_many()`
- `list_candidate_cells()`
- `generate_metric_event_charts()`
- `load_settings()`
- `update_settings()`

初期実装では Web API は挟まない。
設定キャッシュはプロジェクト直下の `.uc_rainfall_settings.json` を利用する。
AI テストでは、通常利用モードとは別にテストモードを設け、機械可読な状態・操作・結果の受け渡しを行う。


## 3. 採用方針

### 3.1 技術選定

- GUI フレームワーク: Tkinter
- 理由:
  - ローカルツールとして十分
  - 既存環境で追加依存が少ない
  - CLI / サービス層をそのまま呼びやすい


### 3.2 操作単位

GUI は以下の4機能を中心に構成する。

- 取り込み
- 候補セル確認
- 面的可視化
- グラフ出力


### 3.3 AI GUI テスト方針

AI エージェントが GUI を実際に操作し、視覚的に評価しやすいように、GUI は以下を満たす。

- GUI の主要入力状態を機械可読なファイルへ保存できる
- GUI 上の操作可能要素を機械可読な形で列挙できる
- AI エージェントからの操作要求をテストモードで受け取れる
- 操作後の状態、ログ、スクリーンショットを保存できる
- 直近の実行ログを外部保存できる
- 直近の出力結果を機械可読な要約として保存できる
- テストモードは通常操作導線と分離できる
- AI がボタン不具合、文言の分かりづらさ、文字切れ、重なり、視認性不足を検知しやすいこと

初期実装では、以下の補助ファイルを想定する。

- `.uc_rainfall_settings.json`
  - 前回入力値の保存
- `.uc_rainfall_gui_context.json`
  - 現在状態のスナップショット
- `.uc_rainfall_gui_widget_tree.json`
  - 画面上の操作可能要素一覧
- `.uc_rainfall_gui_action_request.json`
  - AI からの操作要求
- `.uc_rainfall_gui_action_result.json`
  - 操作結果
- `.uc_rainfall_gui_last_run.json`
  - 直近実行結果の要約
- `.uc_rainfall_gui_log.txt`
  - ログ保存先
- `.uc_rainfall_gui_last_screenshot.png`
  - 直近画面キャプチャ


## 4. 画面構成

初期実装では、単一ウィンドウ内に3つの主要エリアを持つ。

1. 入力エリア
2. 候補セルエリア
3. 実行ログエリア

GUI 起動時には設定キャッシュを読み込み、前回入力値を初期表示へ反映する。
通常モードでは主導線を邪魔しないことを優先し、テストモード時のみ AI テスト用の補助情報を強化する。

AI が GUI を扱いやすくするため、以下の UI 原則を追加する。

- ボタン名やラベル名は重複を避ける
- 主要ボタンには一意な `widget_id` を付与する
- エラーメッセージは入力欄との対応が分かる短い日本語にする
- 文字切れや重なりを検知しやすいように、固定幅に押し込みすぎない
- テーブル列名も意味が一意に分かる名称にする


### 4.1 入力エリア

入力項目:

- DB パス
- 入力パス
  - ZIP
  - 展開済みディレクトリ
  - 複数選択可能
- ポリゴンディレクトリ
  - 任意
  - 未指定時は DB 登録済みポリゴンを使用
- 取り込み用 `dataset_id`
  - 任意
  - 単一入力時のみ有効
  - 入力 ZIP / ディレクトリ名とは異なる識別子を付けたい場合に使う
- グラフ用優先 `dataset_id`
  - 任意
  - 未指定時は位置一致する全データセットを対象にする
  - 指定時は同一位置セルのうち優先データセットとして扱う
- 流域名
- 系列モード
  - `cell`
  - `polygon_sum`
  - `polygon_mean`
  - `polygon_weighted_sum`
  - `polygon_weighted_mean`
- 流域内行
- 流域内列
- 表示開始日時
- 表示終了日時
- 出力ディレクトリ

優先的に前回値を復元する項目:

- DB パス
- ポリゴンディレクトリ
- 入力パス一覧
- 出力ディレクトリ
- グラフ用優先 `dataset_id`
- 流域名
- 系列モード
- 表示開始日時
- 表示終了日時

ボタン:

- `DB初期化`
- `取り込み`
- `候補セル一覧更新`
- `グラフ出力`
- `テスト状態を保存`
- `ログを保存`
- `テストモード切替`


### 4.2 候補セルエリア

候補セル一覧はテーブル形式で表示する。

表示列:

- 流域名
- 流域内行
- 流域内列
- 中心X
- 中心Y
- 重なり率
- データセット数

操作:

- 行選択
- 選択行から `流域内行` / `流域内列` を入力欄へ反映

初期実装ではソートとフィルタは最小限とし、将来拡張とする。


### 4.3 テスト支援操作

表示対象:

- 現在の操作サマリ
  - DB パス
  - 流域名
  - 系列モード
  - 流域内行 / 列
  - 表示期間
- 直近実行結果の要約

操作:

- `テスト状態を保存`
  - `.uc_rainfall_gui_context.json` を出力
- `ウィジェット状態を保存`
  - `.uc_rainfall_gui_widget_tree.json` を出力
- `画面を保存`
  - `.uc_rainfall_gui_last_screenshot.png` を出力
- `ログを保存`
  - `.uc_rainfall_gui_log.txt` を出力
- 実行成功時に `.uc_rainfall_gui_last_run.json` を更新する

通常モードでは、テスト支援操作から処理を開始しない。
テストモードでは、AI からの操作要求を受け取って GUI 操作を実行できる。


### 4.4 実行ログエリア

テキストボックス形式でログを表示する。

表示対象:

- 取り込み開始 / 終了
- ポリゴン読込元
  - ファイル
  - DB
- 重複スキップ
- 不整合エラー
- 候補セル数
- 使用データセット
- 最大イベント抽出結果
- 出力ファイルパス
- 処理段階
  - 初期化
  - 取り込み
  - 候補更新
  - グラフ出力
  - テスト操作


## 5. レイアウト案

推奨レイアウト:

- 上段: ファイル / 実行パラメータ入力
- 中段: 候補セルテーブル
- 下段: ログ

横幅の大きい入力項目:

- DB パス
- 入力パス
- ポリゴンディレクトリ
- 出力ディレクトリ

これらは `Entry + 参照ボタン` とする。
入力パスは複数選択に対応し、選択済み一覧を別ウィジェットで確認できるようにする。


## 6. 画面状態制御

### 6.1 系列モードによる制御

- `cell` の場合
  - `流域内行`
  - `流域内列`
  を有効にする

- `polygon_sum`, `polygon_mean`, `polygon_weighted_sum`, `polygon_weighted_mean` の場合
  - `流域内行`
  - `流域内列`
  を無効にする

### 6.2 入力件数による制御

- 入力パスが 1 件のとき
  - 取り込み用 `dataset_id` を有効にする

- 入力パスが複数件のとき
  - 取り込み用 `dataset_id` を無効にする
  - 自動的に各入力ファイル名 / ディレクトリ名ベースで `dataset_id` を決定する


### 6.2 ポリゴンディレクトリによる制御

- 入力時に `ポリゴンディレクトリ` が空でも実行可能
- ただし DB にポリゴン未登録ならエラー表示

### 6.3 テストモードによる制御

- 通常モードでは、テスト状態保存やログ保存から実行が始まらない
- テストモード時のみ、AI の操作要求を受け付ける
- テストモードでは、各ウィジェットに一意な識別子を付与し外部参照可能にする
- GUI の主要ボタン配置や基本操作を圧迫しない位置に置く


## 7. 操作フロー

### 7.1 初回利用

1. DB パスを指定
2. ポリゴンディレクトリを指定
3. 入力パスを指定
4. 必要なら取り込み用 `dataset_id` を指定
5. `取り込み` を実行
6. `候補セル一覧更新` を実行
7. 候補セルを選択
8. 表示期間と出力先を指定
9. `グラフ出力` を実行


### 7.2 DB 登録済みポリゴン利用

1. DB パスを指定
2. 入力パスのみ指定
3. `取り込み` を実行


### 7.3 既存 DB からのグラフ出力のみ

1. DB パスを指定
2. 流域名を選択
3. `候補セル一覧更新`
4. セルまたは流域集計モードを選択
5. 表示期間と出力先を指定
6. `グラフ出力`


### 7.4 AI GUI テストを使う場合

1. GUI をテストモードで起動する
2. GUI の現在状態を `.uc_rainfall_gui_context.json` と `.uc_rainfall_gui_widget_tree.json` へ出力する
3. AI エージェントがウィジェット一覧と画面状態を参照する
4. AI エージェントが `.uc_rainfall_gui_action_request.json` に操作要求を書く
5. GUI が要求を処理し、`.uc_rainfall_gui_action_result.json` と `.uc_rainfall_gui_last_screenshot.png` を更新する
6. AI エージェントが画面、ログ、出力結果を見て UX を評価する


## 8. イベントハンドラ設計

### 8.1 DB 初期化

呼び出し先:

- `initialize_schema()`

結果:

- 成功時にログ出力
- 失敗時はエラーダイアログ
- 成功時は DB パスを設定キャッシュへ保存


### 8.2 取り込み

呼び出し先:

- 単一入力時:
  - `ingest_uc_rainfall()`
- 複数入力時:
  - `ingest_uc_rainfall_many()`

結果:

- 成功時にログ出力
- 重複完全一致時はスキップログ
- 不整合時はエラーダイアログ
- 成功時は入力関連設定を設定キャッシュへ保存

実行ルール:

- 入力 1 件:
  - 取り込み用 `dataset_id` をそのまま渡せる
- 入力 複数件:
  - `ingest_uc_rainfall_many()` を呼ぶ
  - 各 `dataset_id` は入力名から自動決定する


### 8.3 候補セル一覧更新

呼び出し先:

- `list_candidate_cells()`

結果:

- テーブル再描画
- 候補セル 0 件なら通知


### 8.4 グラフ出力

呼び出し先:

- `generate_metric_event_charts()`

結果:

- 出力ファイル一覧をログ表示
- 失敗時はエラーダイアログ
- 成功時は表示期間・流域・出力先を設定キャッシュへ保存

実行ルール:

- グラフ用優先 `dataset_id` 指定あり:
  - `generate_metric_event_charts()` へ `dataset_id` を渡す
  - 同一位置セルのうち優先データセットとして扱う

- グラフ用優先 `dataset_id` 指定なし:
  - `dataset_id=None` で `generate_metric_event_charts()` を呼ぶ
  - 位置一致する全データセットを対象にする


### 8.5 テスト状態保存

呼び出し先:

- GUI 状態出力処理

結果:

- `.uc_rainfall_gui_context.json` を更新
- 成功時にログ出力


### 8.6 ウィジェット状態保存

呼び出し先:

- ウィジェット状態出力処理

結果:

- `.uc_rainfall_gui_widget_tree.json` を更新
- 各 widget の `widget_id`, 種別, 表示名, 値, 状態, 位置, サイズを保存する


### 8.7 画面保存

呼び出し先:

- 画面キャプチャ出力処理

結果:

- `.uc_rainfall_gui_last_screenshot.png` を更新
- AI が視覚確認できるようにする


### 8.8 ログ保存

呼び出し先:

- ログ保存処理

結果:

- `.uc_rainfall_gui_log.txt` を出力する
- 直近の GUI ログを外部確認できるようにする


### 8.9 テストモード操作要求処理

呼び出し先:

- テストモード操作要求処理

結果:

- `.uc_rainfall_gui_action_request.json` を読み込む
- 対象 widget と操作内容を解決する
- 操作結果を `.uc_rainfall_gui_action_result.json` に保存する
- 必要に応じて状態・ログ・スクリーンショットを更新する


## 9. バリデーション

最低限の入力検証:

- DB パスが空でない
- 入力パスが存在する
  - 取り込み時のみ
- 複数入力時は取り込み用 `dataset_id` を空にする
- グラフ出力時は `polygon_name` が選択済み
- `cell` モードでは `流域内行` / `流域内列` が入力済み
- `view_start <= view_end`
- 出力ディレクトリが指定されている
  - グラフ出力時
- テスト状態保存先やログ保存先は書き込み可能であること
- テストモードでは未知の widget_id や未知の操作種別を拒否する
- テストモードでは GUI 対象外の危険操作を拒否する


## 10. 候補セルテーブルの実装方針

Tkinter の `ttk.Treeview` を第一候補とする。

理由:

- 標準で利用可能
- テーブル表示に十分
- 行選択イベントを扱いやすい

将来拡張:

- 列ソート
- CSV エクスポート
- ダブルクリックでセル選択


## 11. ログの扱い

サービス層は既に日本語ログを出すため、GUI では以下のいずれかを採る。

- 標準 logging を GUI テキストへ転送する
- GUI 側で service 呼び出し結果を補足して追記する

初期実装では、logging ハンドラを追加して GUI テキストへ流す方式を第一候補とする。
必要に応じて、ログ保存時に処理段階と時刻を付与する。
テストモード操作では、対象 widget_id と実行結果もログへ残す。


## 12. スレッド / 応答性

長時間処理:

- 取り込み
- グラフ出力

これらは UI スレッドを塞ぎやすい。

初期実装では、少なくとも以下を行う。

- 実行中はボタンを無効化
- カーソルまたは状態ラベルで実行中を示す

将来拡張:

- バックグラウンドスレッド実行
- 進捗表示

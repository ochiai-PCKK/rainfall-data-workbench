# UC Rainfall ZIP Flow JAXAモード 調査メモ

## 1. 文書の位置づけ

本書は要件定義ではなく、JAXAモードの採用可否を判断するための調査メモである。  
現時点では「実装着手しない」前提で、論点と再開条件を残す。

## 2. 背景

- ZIPモード/Excelモードに加え、JAXA APIから直接取得して同様のグラフを作る案を検討した。
- 取得範囲は既存ポリゴンのBBox + バッファ、取得後に重み付き sum/mean 集計する方針を想定した。

## 3. 調査結果（現時点）

- `jaxa-earth` v0.1.4 で参照できる GSMaP 系コレクションは、実運用上 daily / half-monthly / monthly 系が中心。
- 本検討で想定した「1時間間隔の時系列を直接取得して既存ZIPモードと同等処理」は、そのままでは成立しにくい。
- 既存 `JAXA.EORC_GSMaP_standard.Gauge.00Z-23Z.v6_daily` の開始時刻は STAC 上で `2000-03-01T00:00:00Z`。

## 4. 判断

- JAXAモードは、現時点ではプロダクト要件に昇格しない。
- `requirements.md` 系の必須要件には追加しない。
- 本件は「将来検討項目（調査継続）」として扱う。

## 5. 再開条件

以下のいずれかを満たした時点で再評価する。

1. 必要な時間解像度（少なくとも運用要件を満たす粒度）のデータソースを確保できる。
2. 既存 ZIP/Excel と同等の比較可能性を担保できる内部フォーマットが確定する。
3. API利用制約（レート、認証、可用性）を運用設計として吸収できる。

## 6. メモ（設計観点）

- 将来再開する場合も、入力アダプタを増やして描画前の内部モデルは共通化する方針が有効。
- UIは JST 入力、APIは UTC リクエスト、応答は JST 正規化の流れを統一する。
- 期間窓（3日/5日、左寄せ/中央/右寄せ）は既存ロジックをそのまま使う。
